# auto_ship feature 设计（自动发货）

- 日期：2026-05-28
- 状态：设计已确认，待转 writing-plans
- feature：`auto_ship`「自动发货」，order=6
- 同页参考：`packing_label`（同页 `shipping-list`、同 rowspan 表格结构，但工作在不同 in-page tab）
- 跨 tab 编排参考：`create_purchase_order`（本 feature 不需要 background 编排，理由见 §4）

## 1. 背景与问题

跨境卖货后台发货单列表页 `https://seller.kuajingmaihuo.com/main/order-manager/shipping-list`，在**「待装箱发货」in-page tab** 下，用户需要对每个**非本地仓**的发货单逐个走一套固定流程：勾选行 → 打印商品打包标签（走"先发货后打印"）→ 等包裹号刷新 → 批量装箱发货 → 在编辑页填「箱子和袋子 + 1 箱」→ 确认发货。一个发货单要点 6~8 次、跨 3 个弹窗 + 1 个编辑页，几十个单子重复，极其费时且容易点错/漏单。

「确认发货」是**真实出货的不可逆动作**，所以这一步必须有人工门控，不能默认全自动。

## 2. 目标 / 非目标

**目标**：
- 在「待装箱发货」tab 一键遍历所有发货单，对**非本地仓**单自动走完整流程到「确认发货」前
- 「确认发货」由 Hub 开关 + 逐单弹窗门控（默认人工逐单决定）
- 本地仓（发货仓库 = `化州中正科技`）单自动识别并跳过
- 包裹号已存在的单跳过打印步、直奔装箱发货
- 无人值守容错：单步失败记录原因 + 跳过该单 + 继续下一单
- 防脏数据：流程自探测当前 in-page tab，不在「待装箱发货」主动切回；已发货单若滞留则切 tab 刷新后台数据再继续

**非目标**：
- 不做整页 reload 续跑（确认发货是 SPA 切 tab，不整页刷新，循环态可驻留内存——见 §10）
- 不捕获/保存 PDF（"先发货后打印"是延迟打印，本 feature 不碰打印产物，无 native host）
- 不处理「待仓库收货」tab 的任何操作（那是 packing_label 的领域）
- 不做跨浏览器 tab 编排（全流程在单个浏览器 tab 内）

## 3. 需求与已确认决策

| 项 | 决策 |
|----|------|
| 确认发货门控 | Hub 开关「自动确认发货」默认 **OFF**：OFF → 每单走到确认发货步骤弹自定义确认框，用户逐单决定（确认→点确认发货；取消→关编辑页跳过该单）；ON → 不弹窗，每单自动点确认发货 |
| 单步失败处理 | **记录「发货单号 + 失败步骤 + 原因」+ 标记该单 processed + 继续下一单**（无人值守，不中断整批） |
| 行加载/遍历 | 表格虚拟滚动：**每轮重扫活表格 + 按发货单号去重 + processed-set**；不一次性快照全部行（节点会被回收 + 表格运行中会变） |
| 确认发货后页面行为 | **SPA 切到「待仓库收货」tab，不整页 reload**；流程自探测当前 tab，不在「待装箱发货」主动切回 |
| 本地仓识别 | 发货仓库文本 == `化州中正科技` → 本地仓 → 跳过 |
| 包裹号已存在 | 该单「包裹号」列已有值 → 跳过打印步，直奔批量装箱发货 |
| feature 归属 | 纯 content-script 单页编排；无 background 标记段、无 native host、无 extra_content_scripts；permissions 仅 `storage` |

## 4. 架构：纯 content-script 单页编排

```
features/auto_ship/content/index.js  (isolated, document_idle)
  • registerFeature + Hub view：开始/停止 + 「自动确认发货」开关 + 进度/汇总
  • 主循环（驱动层）：探测 in-page tab → 滚动重扫活表格 → 去重 → 取下一个未处理单
  • 单发货单状态机：读仓库 → 打印步 → 等包裹号 → 装箱发货 → 编辑页填写 → 门控确认发货
  • 自定义确认框（OFF 模式逐单弹）：可拖动 modal，确认/取消
```

**为什么不用 background 编排**：`create_purchase_order` 把编排放 background 是因为它**跨 3 个域、跨多个浏览器 tab**，只有 service worker 能跨 tab 存活、用 `chrome.storage` 当单一状态源。本 feature 全流程在**一个浏览器 tab 内**、SPA 不整页 reload，循环态可直接驻留 content script 内存。引入 background 是过度设计（违反 coding-rules「不发明无收益的抽象层」）。

**唯一进 storage 的**：Hub「自动确认发货」开关（用户设置，跨会话保留，键名 `auto_ship_auto_confirm`）。循环运行态（processed 集合、进度、失败明细）在内存。

## 5. 单发货单状态机（每行流程）

对一个发货单号 `S`（已在主循环中定位到其行）：

```
1. 读 S 的「发货信息」列 → 发货仓库文本
   ├ == 化州中正科技 (本地仓) → 标记 processed, return(skipped:本地仓)
   └ 否 → 继续

2. 读 S 的「包裹号」列
   ├ 已有包裹号 → 跳过打印步, 直奔第 3 步
   └ 无 → 勾选 S 分组 checkbox (行高亮) → 点「打印商品打包标签」
        → 弹窗「确认打印商品打包标签」→ 点下划线「先发货后打印」
        → 小弹窗 → 点「确认」
        → 轮询 S 的「包裹号」列直到出现 (带超时; 超时前可切 tab 刷新一次重读)

3. 确保 S 分组 checkbox 选中 → 点「批量装箱发货」(选中才高亮可点)
   → 弹窗确认 → 点「去装箱发货」→ 出 inner 编辑页

4. 编辑页填写 (每项写后读校验, 对齐项目「写后读」铁律):
   - 包装方式 → 点选「箱子和袋子」→ 回读确认
   - 发货总箱/包数 → 文本框写「1」→ 回读 value == 1

5. 确认发货门控:
   ├ 开关 ON → 自动点「确认发货」
   ├ 开关 OFF → 弹自定义确认框:
   │   ├ 用户确认 → 点「确认发货」
   │   └ 用户取消 → 关闭 inner 编辑页 (留在待装箱发货)
   └ 标记 S processed

6. 若点了确认发货 → SPA 切到「待仓库收货」
   → 主循环下一轮探测到不在待装箱发货 → 主动切回
   → 校验 S 已从待装箱发货消失; 未消失(脏数据) → 切 tab 刷新重扫
```

任一步抛错 → catch → 记录 `{发货单号, 失败步骤, 错误类别+原因}` → 标记 processed → 继续下一单。

## 6. 主循环（驱动层）

```
processed = Set<发货单号>   (内存)
fails = []                  (内存, 汇总用)
开局: 轻量滚动全扫一次 → 拿总单数 N (发货单号列表), 进度显示 i/N

loop (用户未点停止 且 仍有未处理单):
  ① 探测当前 in-page tab; 不是「待装箱发货」→ 点 tab 切回
  ② 滚动扫描活表格 → 枚举当前所有发货单号 (去重)
  ③ pick = 第一个不在 processed 的发货单号
  ④ pick 为空:
       切 tab 刷新 + 再扫一次确认确实没有 → break
  ⑤ 定位 pick 行 → 跑单发货单状态机 → processed.add(pick) → 继续 loop

结束: 面板汇总「处理 X 单 / 跳过本地仓 Y / 失败 Z (明细)」
```

**滞留两类"已处理但留在原地"的行**（必须靠 processed 去重，否则死循环）：
- 本地仓单（跳过后仍在表里）；
- 非本地仓但用户取消确认发货的单（编辑页关掉、留在原地，且包裹号可能已生成）。

## 7. DOM 选择器（实现前必须先 dump，本节为占位）

> 本 feature 工作在**「待装箱发货」tab**，与 `packing_label`（待仓库收货 tab）按钮行为不同，**不能直接复用其 selector**。实现前我提供 console 脚本，由用户在真实页面 dump 5 处 DOM 存入 `features/auto_ship/samples/`，再据真实 DOM 定 selector（项目铁律：取数/选择器类一律先 dump 再写，`create_purchase_order` 踩过多次）。

待 dump 的 5 处：

| 编号 | 目标 | 关心的字段 |
|------|------|-----------|
| ① | 表格几行（含 checkbox） | 发货单号列、发货信息列(发货仓库)、包裹号列、操作列(打印商品打包标签/批量装箱发货按钮)、分组 checkbox 结构、虚拟滚动容器 |
| ② | 「确认打印商品打包标签」弹窗 | 下划线「先发货后打印」按钮定位 |
| ③ | 「先发货后打印」后的小确认弹窗 | 「确认」按钮定位 |
| ④ | 「批量装箱发货」确认弹窗 | 「去装箱发货」按钮定位 |
| ⑤ | inner 编辑页 | 「包装方式」选择器(箱子和袋子选项)、「发货总箱/包数」输入框、「确认发货」按钮 |

通用 Temu Beast UI 约定（沿用 packing_label / create_purchase_order 经验）：优先 `data-testid` + 文字匹配；类名前缀模糊匹配（`IPT_`/`BTN_` 等）；禁用带 hash 的 class（随版本漂移）；in-page tab、弹窗、选项结构均需 dump 确认。

## 8. Hub UI 与门控

Hub view（feature 面板）元素：

- **「自动确认发货」开关**：默认 OFF，状态存 `chrome.storage.local['auto_ship_auto_confirm']`，跨会话保留。OFF=逐单弹窗确认；ON=全自动点确认发货。
- **「开始」/「停止」按钮**：开始启动主循环；停止设置中断标志，当前单跑完后退出循环。
- **进度区**：`处理中 i/N`、当前发货单号、当前步骤。
- **汇总区**：结束后显示「处理 X / 跳过本地仓 Y / 失败 Z」+ 失败明细列表（发货单号 + 步骤 + 原因）。

**OFF 模式逐单确认框**：可拖动 modal（复用 core utils `makeDraggable`），展示当前发货单号 + 「确认发货 / 取消」两个按钮。确认框 Promise 直到用户点击才 resolve（不超时、不 retry，避免打断用户核对——对齐 create_purchase_order 半自动确认的踩坑）。

## 9. 错误处理（分层文案，遵循 debugging-rules 错误分层铁律）

| 类别 | 例 | 文案模板 |
|------|----|---------|
| 读取/选择器故障 | checkbox/按钮/弹窗没找到 | 「读取失败：未找到 X（发货单 S，步骤 N）」 |
| 数据校验 | 包装方式/箱数填后回读不符 | 「数据校验：X 填写后不符，期望「箱子和袋子」/「1」实际「Y」」 |
| 业务/超时 | 包裹号超时未刷新 | 「业务：包裹号超时未生成（发货单 S）」 |

三类不混用同一文案（避免把数据问题误诊成读取问题）。单单失败计入 `fails`，不中断整批，结尾汇总。

## 10. 状态与持久化

| 状态 | 存放 | 理由 |
|------|------|------|
| 「自动确认发货」开关 | `chrome.storage.local` | 用户设置，跨会话保留 |
| processed 集合 / 进度 / fails | content script 内存 | SPA 不整页 reload，内存够用；F5 会重头开始（半自动有人盯着，可接受） |

**写后读校验**（项目铁律）：编辑页每个控件填完回读实际值确认 == 期望，不符报「数据校验」层错误。包装方式（select 类）读选中项文本；箱数（input）读 `value`。

**防脏数据多信号兜底**（沿用 price_declare「判完成多信号」+ 主动切 tab 刷新经验）：
- 包裹号轮询：点完小弹窗确认后轮询包裹号列，超时前切 tab 刷新一次再读；
- 已发货校验：确认发货后，下一轮主循环校验该单已离开待装箱发货；滞留则切 tab 刷新重扫，不据滞留数据继续操作。

## 11. 边界 / 已知风险

- 「批量装箱发货」按钮选中行才高亮可点——点前必须确保 checkbox 已勾且按钮非 disabled，否则报读取/业务类错。
- 一个发货单号是否对应一个分组 checkbox（rowspan）需 dump 确认；若一个发货单跨多个物流分组，选中逻辑需调整（dump 后定）。
- 虚拟滚动行数未知；开局全扫与每轮重扫都需滚动到底，超大列表性能待实测（先按几十单规模设计）。
- ON 全自动模式连续发真实货，用户须自行确认批次正确——默认 OFF 即为此防线。
- in-page tab 切换 + 刷新机制需 dump 确认（点 tab 元素是否触发后台重新拉数，还是另有刷新按钮）。

## 12. 测试

- **纯函数**（发货仓库判定 `是否本地仓`、包裹号是否存在判定、发货单号提取/去重、失败明细聚合）→ 抽 `auto_ship-logic.js` 双模式模块，`node --test`，Linux 可跑。
- **DOM 编排 / 状态机 / 门控 / 切 tab** → Windows + 真实页面手动验证；实现顺序：先 dump DOM → 先验证「单个发货单走通一遍」→ 再接主循环批量。

## 13. Open Issues（写 plan / 实现时定）

1. **发货单号 ↔ 分组 checkbox 的对应关系**：一个发货单是否恰好一个 checkbox（rowspan 分组），dump ① 后确认；影响选中与定位逻辑。
2. **in-page tab 刷新方式**：点 tab 是否重新拉后台数据，是否有显式刷新按钮；dump 后定切 tab 刷新的具体动作。
3. **包裹号轮询超时阈值 + 切 tab 刷新时机**：点完"先发货后打印"小弹窗后，包裹号生成需多久；超时多少触发切 tab 刷新。
4. **包裹号"已存在"判定**：包裹号列空值的真实表现（空字符串 / `-` / 占位元素），dump ① 后定。

## 14. 不改动的部分

core API、service-worker、native_host、其它 feature 均不动。本 feature 自治在 `features/auto_ship/`，permissions 仅 `storage`。
